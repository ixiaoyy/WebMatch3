# 可玩界面与交互反馈

## Goal

把核心规则引擎呈现为一套在手机、平板和桌面浏览器中都清晰、顺畅且可访问的可玩界面，不在本任务内锁定最终品牌主题。

## Background

- 核心引擎已提供不可变棋盘、显式随机源和按顺序排列的 `CascadeRound`。
- 本任务先以“练习模式”承载可玩交互；计分、有限步数、目标和胜负由 `07-17-session-progression` 接入。
- HUD 与结果层使用可空、类型化的展示契约，未接入的局内值显示为明确的练习状态，不伪造计分结果。

## Requirements

### R1. 信息架构

- 单页至少包含：游戏标题/入口、局内 HUD、棋盘、操作提示、重新开始、玩法说明和结果层。
- HUD 展示当前分数、目标分、剩余步数和连击信息；布局不遮挡棋盘。
- 初次进入的说明保持简短，用户可关闭并直接开始。

### R2. 棋盘呈现

- 棋盘始终保持规则网格和稳定纵横比，尺寸变化不能造成棋子重叠或点击区域错位。
- 每种棋子同时使用颜色与形状/纹理区分，兼顾色觉差异和小屏辨识度。
- 棋子组件只接收展示所需状态，不能自行执行匹配或计分逻辑。
- 选中态、建议下一步（若后续启用）和不可交互态视觉语义互不混淆。

### R3. 输入方式

- 鼠标/触控：先选一个棋子，再选相邻棋子完成交换；选择同一棋子可取消；选择非相邻棋子时切换当前选择。
- 键盘：焦点可在棋盘中移动，并能选择、交换、取消；操作不要求使用鼠标。
- 动画和结算期间禁用冲突输入，但页面其他非冲突控制的可用性需要明确。
- 重新开始在进行中的对局上触发时，需要防止误触丢失进度，可采用二次确认或撤销语义。

### R4. 动效与反馈

- 有效交换、无效复位、匹配消除、坠落补充、连锁、洗牌和结果出现具有可区分反馈。
- 动效顺序必须反映核心引擎返回的结算轮次，不能为视觉效果跳过或重排规则状态。
- 动画时长以不拖慢操作为原则，单次普通交换到可再次操作的等待目标不超过 1 秒。
- 减少动态模式下缩短或移除位移、缩放和闪烁，但保留文本/静态状态反馈。

### R5. 响应式

- 320–430px 手机宽度下，棋盘完整显示、无横向滚动、核心操作触达方便。
- 平板和桌面允许 HUD 与棋盘并排，但不能把手机布局简单等比放大。
- 横屏低高度设备需要保证棋盘和主要控制仍可访问，可允许页面纵向滚动。
- 处理浏览器安全区和动态视口高度，结果层与控制按钮不得被地址栏或底部区域遮挡。

### R6. 可访问性

- 棋盘具有可理解的网格语义；棋子名称包含类型、行列位置和选中状态。
- 状态区域以不过度打扰的方式播报无效交换、连击、洗牌和胜负。
- 所有按钮有可读名称和清晰 `focus-visible`，文本对比度满足 WCAG AA。
- 不使用高频闪烁；可交互状态不只依赖 hover。

### R7. 视觉边界

- 本任务先建立视觉令牌与可换肤的棋子表现，不把暂定工程名当作正式品牌。
- 最终主题、Logo、插画和音频需要产品确认后再固化。
- 不复制 ParallelLines 的 Logo、配色或社区视觉语言。

## Acceptance Criteria

- [x] AC1：用户可用鼠标、触控和键盘完成选择、交换、取消和重新开始。
- [x] AC2：无效交换会复位且有明确反馈；结算期间快速连续输入不会破坏棋盘状态。
- [x] AC3：消除、坠落、连锁和洗牌按引擎轮次依序呈现，动画结束后视觉棋盘与规则棋盘一致。
- [x] AC4：320×568、390×844、768×1024 和 1440×900 视口无横向溢出、遮挡或错位。
- [x] AC5：棋子不只依赖颜色区分，文本、焦点和关键控件对比度满足 WCAG AA。
- [x] AC6：`prefers-reduced-motion: reduce` 下可完整游玩且无高幅位移动画。
- [x] AC7：页面中没有 ParallelLines 品牌资产，也没有未经确认的正式产品名或 Logo。

## Dependencies

- 依赖 `07-17-project-foundation`。
- 依赖 `07-17-match3-core-engine` 的棋盘状态与结构化结算结果契约。
- 与 `07-17-session-progression` 通过 `GameHudState`、`GameResultState` 和新局回调对齐；本任务不实现扣步、计分或胜负判断。

## Out of Scope

- 最终品牌命名、营销页和品牌插画。
- 音效、背景音乐和震动反馈。
- 特殊棋子、关卡地图和教程关。
