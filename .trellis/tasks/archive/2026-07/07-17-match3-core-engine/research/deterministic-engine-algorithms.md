# 确定性三消引擎算法调研

## 约束

- 公开棋盘始终稳定且不可变，单元格均包含棋子。
- 匹配只考虑横向或纵向连续 3 个及以上同类型棋子。
- 所有随机路径必须可注入、可复现并有尝试上限。
- UI 需要每轮消除、坠落、补充和洗牌的结构化变化，而不是读取引擎中间态。

## 采用算法

### 匹配扫描

分别逐行和逐列做 run-length 扫描。长度达到 3 时产生匹配组；再按 `row:column` 合并唯一坐标。这样既保留横纵组供计分，也保证 T/L/十字交叉公共棋子只删除一次。

### 合法移动

只枚举每格的右邻与下邻，交换后运行匹配扫描。稳定棋盘原本没有匹配，因此交换后存在任意匹配即可判定为合法移动。枚举不会产生重复坐标对。

### 初始生成

按行优先填充，每格从随机起始类型开始循环候选，跳过会与左侧两个或上方两个形成三连的类型。生成后检查至少一个合法移动；失败则重试到上限。

兜底棋盘在左上角植入：

```text
A B A
C A C
```

交换 `(0,1)` 与 `(1,1)` 必然形成顶行 `A A A`。其余格继续用“不形成即时三连”的确定性候选填充，因此兜底结果同时满足“无初始匹配”和“至少一步可走”。

### 消除与坠落

每轮先合并匹配坐标并标空，再按列从底向上收集幸存棋子，保持其相对顺序并移动到底部。顶部空位由随机源补充。每个变化包含棋子、起点和终点；新棋子的起点使用负行号，便于 UI 从棋盘上方播放进入动画。

### 洗牌

优先对现有棋子做有限次 Fisher–Yates 洗牌，保留棋子 ID 与类型数量。只有结果同时“无已有匹配且至少一步可走”才接受。达到上限后调用确定性生成兜底并显式返回 `rebuilt`，让上层知道类型数量可能已重建。

## 有限性与错误

- 配置、随机值和棋盘形状先验证。
- 初始生成、洗牌和连锁分别使用独立上限。
- 随机源必须返回有限的 `[0, 1)` 数值。
- 达到最大连锁轮次后若棋盘仍有匹配，抛出带错误码的诊断错误。
