# 强化核心玩法与失败反馈：技术设计

## 设计目标

用三个可独立交付的增量建立完整反馈闭环：先让失败与成长规则成立，再让每局布局真正变化，最后统一搜索、选择和奖励的触感。规则保持纯函数，Vue 控制时间编排，组件只负责表现。

## 边界与数据流

```text
输入（指针 / 触摸 / 键盘）
  → FishField / FishPiece
  → ambient-controller
  → engine 纯状态转换
  → 立即持久化稳定状态
  → UI 临时反馈预览
  → 反馈结束后恢复可操作状态
```

- `engine/` 拥有选鱼、三消、失败重开、关卡生成和可解性契约。
- `session/` 只验证和恢复稳定快照，不保存动画中间帧。
- `ambient-controller.ts` 拥有反馈状态、计时器、离开/恢复处理和稳定状态持久化。
- Vue 组件使用明确的 `data-*` 状态渲染动效，不重新实现规则。

## 增量一：失败与植物成长

### 失败状态转换

以 `lost` 取代 `recovery-needed`：

- 第七条鱼进入托盘且未触发普通三消或猫咪抵扣时，`selectPiece` 返回失败结果。
- 失败结果同时携带七条鱼的只读失败预览，以及已经重置好的稳定状态。
- 重置状态通过 `createLevelState(1, previous.clearCount, previous.nextPieceId, random)` 创建，因此等级回到 1，托盘和 `fed` 清空，植物经验与 ID 单调性保留。
- controller 立即保存重置后的稳定状态，再用失败预览播放 1–1.5 秒反馈。页面在动画期间失焦或关闭时，不会把满盘中间态保存成下一次启动状态。
- 失败开始时撤销猫咪守护目标并复位猫咪状态；反馈结束后自动开放第一局，不出现弹窗或按钮。

### 植物经验曲线

- `clearCount` 继续作为植物累计经验的唯一来源，失败不修改它。
- 将阶梯式 `getGrowthPercent` 改为分段连续曲线：第一次清除即产生可见增长，12 次清除到达第一个明确节点。
- 后续节点逐渐拉长，开花、结果、成熟仍同时受累计清除量与种植天数约束。
- 旧玩家的 `clearCount` 不迁移、不回退；新曲线直接映射已有累计值。

## 增量二：受约束随机生成

### 位置

- 用随机候选点替换固定 `SCATTER_POINTS` / `CLUSTER_ANCHORS`。
- 候选点必须位于规范化安全区域，排除桌面右上方猫/花盆主体区，并给底部托盘留出投影余量。
- 使用有上限的拒绝采样控制中心点最小距离、允许的重叠比例和整体覆盖范围；达到重试上限时使用同一随机种子可复现的安全回退布局，禁止无限循环。
- 保留“开局附近可发现匹配”的生成后校验，但不把三个同类重新堆在同一位置。

### 种类、层级与可解性

- 保留完整三元组计数和最近加入的跨空间组种类调度，确保每种剩余鱼数量始终是 3 的倍数。
- 为各空间组生成平衡的层级配额后使用同一随机源洗牌，避免层级按数组块固定排列。
- 当前规则允许所有剩余鱼被选择，因此层级是视觉遮挡关系而不是硬性解锁条件；完整解仍由种类计数和三元组调度保证。
- 最终位置确定后统一重算 `blockerIds`，重叠判定使用旋转无关的保守圆形/方形足迹，避免 360° 旋转造成视觉与命中关系漂移。

### 旋转

- 每条鱼从 `[0, 360)` 独立采样角度，不排除倒立或竖直方向。
- 旋转继续属于稳定游戏状态并进入快照；相同种子产生完全一致的位置、层级、角度和种类。

## 增量三：统一交互反馈

采用“安静、触觉、发光”的统一运动语言；直接操作一般为 150–250ms，状态奖励可延长至 480–700ms，失败序列为 1–1.5 秒。使用指数型 ease-out，不使用弹簧、赌场粒子或全屏闪烁。

| 事件 | 首轮反馈 |
| --- | --- |
| 首次进入 | 聚光灯自动扫向附近鱼，目标鱼依次轻抬，第一格托盘短暂呼吸；首次操作立即取消引导 |
| 搜索显影 | 光圈位置更清楚，鱼从桌面材质中显出并按层级产生轻微高度/阴影差 |
| 选择入盘 | 原位鱼快速收起，目标托盘格产生落位、压感和短暂高光；周围重叠鱼重新沉降 |
| 托盘 5/6 格 | 槽位逐步收紧并产生克制的警示色/呼吸，不增加 HUD 文案 |
| 三消 | 三条鱼聚拢融化，局部气泡/光晕后植物产生一次永久生长和短暂回应 |
| 满盘失败 | 第七格落位后托盘明显受压，场景短暂降亮、鱼群散开/沉下、猫咪作出反应，随后自动重开 |
| 清场/新局 | 旧场景完全收束后，新鱼按随机层级短暂错峰落定，不做长时间入场表演 |
| 猫咪寻鱼 | 眼神/身体朝向、移动和独立引导光保持连续因果关系 |
| 喂鱼 | 拖放目标高亮、接受时鱼收向猫咪并切换进食姿态，拒绝时回位且给出可辨状态 |

- 第一轮优先扩展现有组件状态和 CSS，不引入动画依赖。
- 若现有猫咪姿态不足以表达失败，只新增一张风格一致的静态失败姿态；其余高级轨迹与更多角色帧作为后续增量。
- `prefers-reduced-motion` 使用即时位置变化、颜色、轮廓和交叉淡化保留全部状态语义。

## 首次引导

- 仅在加载到真正未操作的初始状态时运行；已有托盘、喂食、猫咪守护目标或植物经验时不出现。
- 不新增持久化字段：玩家在引导完成前刷新时允许再次播放；完成第一次选择后现有快照自然阻止再次触发。
- 引导不锁定输入，任意指针、触摸、键盘或猫咪操作都会立即接管并停止自动演示。

## 持久化与兼容

- 游戏状态结构无需新增字段，优先保持快照版本 3。
- 旧快照中的固定布局继续按原样恢复，直到玩家清场或失败后才使用新生成器，避免升级时擅自改写正在进行的局面。
- `PRODUCT.md` 需要同步修改“永不失败”的旧产品原则：允许安静、自动恢复的满盘失败，但仍禁止单独失败页面和惩罚性元进度回退。

## 风险与控制

- **随机布局退化**：有限拒绝采样、确定性回退和多种子属性测试。
- **失败动画被页面生命周期打断**：先持久化重置状态，动画只使用预览数据。
- **反馈状态互相覆盖**：controller 统一拥有互斥反馈枚举和计时器清理，不在多个组件各自计时。
- **范围膨胀**：三个增量分别验证；高级飞行动画、新音效包和逐帧角色动画不阻塞首轮完成。
- **随机旋转误点**：以稳定命中容器和旋转无关重叠足迹验证视觉/交互一致性。

## 回滚

- 三个增量分别形成逻辑提交边界。
- 若新生成器出现退化，可单独恢复旧生成策略而保留失败和成长规则。
- 若某项新反馈影响输入，可关闭对应 `data-*` 表现状态，不回滚纯规则状态。
